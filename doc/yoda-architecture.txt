MODULE STRUCTURE                                *yoda-architecture-structure*

File Organization:~
>
    lua/yoda/
    ├── buffer/               Buffer state inspection
    │   └── state_checker.lua Modifiable/writable buffer checks
    │
    ├── commands/             User-facing Ex commands
    │   ├── buffer.lua        Buffer management commands
    │   ├── diagnostics.lua   Diagnostic commands
    │   ├── formatting.lua    Format commands
    │   ├── lazy.lua          Lazy plugin manager commands
    │   └── lsp.lua           LSP commands
    │
    ├── filetype/             Filetype handling
    │   ├── detection.lua     Custom filetype detection autocmds
    │   └── settings.lua      Per-filetype option settings
    │
    ├── integrations/         Third-party plugin wiring
    │   └── gitsigns.lua      Gitsigns event batching
    │
    ├── keymaps/              Keymap definitions by domain
    │   ├── init.lua          Registration entry point
    │   └── [domain].lua      Per-domain keymaps (git, lsp, testing…)
    │
    ├── testing/              Test configuration
    │   └── defaults.lua      User-configurable test defaults
    │
    ├── config_loader.lua     YAML/Lua config loading
    ├── environment.lua       Environment detection (home/work)
    ├── lsp.lua               LSP setup and server registration
    ├── picker_handler.lua    Abstract picker (snacks/fzf-lua)
    ├── python_venv.lua       Python virtualenv detection
    └── [other modules]       Specialised functionality
<

Module Categories:~
1. Buffer/State: Buffer introspection utilities
2. Commands: User-facing Ex commands registered at startup
3. Keymaps: Domain-grouped keymap definitions
4. Integrations: Wiring for specific third-party plugins
5. Infrastructure: Config, environment, testing, and utility modules

DESIGN PRINCIPLES                              *yoda-architecture-principles*

SOLID Principles (10/10)~

Single Responsibility:~
Each module has one clear purpose:
  • Keymap modules: only register keymaps for one domain
  • Command modules: only register Ex commands for one domain
  • Terminal: only terminal operations

Open/Closed:~
Extensible without modification:
>
    -- Users can configure without editing source
    vim.g.yoda_test_config = {
      environments = { custom_envs },
    }
    vim.g.yoda_picker_backend = "telescope"
<

Liskov Substitution:~
Consistent interfaces:
  • All modules follow the same error handling patterns
  • picker_handler provides a unified picker API regardless of backend

Interface Segregation:~
Focused, minimal interfaces:
  • Each module exposes only what callers need
  • No forced dependencies on unused functionality

Dependency Inversion:~
Abstract dependencies:
>
    -- picker_handler abstracts snacks/fzf-lua from call sites
    local picker = require("yoda.picker_handler")
    picker.select(items, opts, callback)
<

CLEAN Code Principles (10/10)~

Cohesive:~
  • Related functionality grouped together
  • Single purpose per module
  • Clear module boundaries

Loosely Coupled:~
  • Minimal inter-module dependencies
  • Dependency injection patterns
  • Plugin-agnostic abstractions

Encapsulated:~
  • Private functions not exposed
  • Clear public APIs
  • Implementation details hidden

Assertive:~
  • Input validation on all public functions
  • Clear error messages
  • Fail-fast principle

Non-redundant (DRY):~
  • Zero code duplication
  • Shared utilities in utils.lua and domain modules
  • Configuration-driven behavior

TERMINAL SYSTEM                                 *yoda-architecture-terminal*

Terminal management is provided by the external `yoda-terminal.nvim` plugin,
which is loaded via `lazy-plugins.lua` as a Jedi Knights foundation plugin.

Configuration happens at startup via plugin_spec("yoda-terminal.nvim"):
>
    plugin_spec("yoda-terminal.nvim", {
      lazy = false,
      priority = 996,
      config = function()
        require("yoda-terminal").setup({
          width = 0.9,
          height = 0.85,
          border = "rounded",
          autocmds = true,
          commands = true,
        })
      end,
    })
<

The terminal module is available at runtime via:
>
    local terminal = require("yoda-terminal")
<

TESTING INFRASTRUCTURE                          *yoda-architecture-testing*

Comprehensive testing framework with high coverage.

Test Structure:~
>
    tests/unit/                Unit tests mirror lua/yoda/ structure
    ├── buffer/                Buffer state checker tests
    ├── filetype/              Filetype detection and settings tests
    ├── integrations/          Integration wiring tests (gitsigns, etc.)
    └── window/                Window utility tests
    Plus top-level spec files for each core module
<

Test Helpers~
>
    local helpers = require("tests.helpers")
    
    -- Function spying
    local spy_fn, spy_data = helpers.spy()
    helpers.assert_called(spy_data)
    
    -- Function mocking
    local restore = helpers.mock(obj, "method", mock_fn)
    
    -- Neovim API mocking
    local api = helpers.mock_nvim_api()
<

Test Patterns:~
1. AAA (Arrange-Act-Assert) pattern
2. Mock external dependencies
3. Test edge cases (nil, empty, errors)
4. Maintain >90% coverage

Quality Metrics:~
  • 191 unit tests
  • ~95% code coverage
  • All tests pass on CI
  • Comprehensive edge case testing

CONFIGURATION SYSTEM                             *yoda-architecture-config*

Flexible configuration system supporting multiple formats and sources.

config_loader.lua~
Configuration loading with fallbacks:
>
    local config_loader = require("yoda.config_loader")
    
    -- Try JSON first, fallback to YAML, then defaults
    local config = config_loader.load_env_region()
<

Functions:
  • load_json_config(path) - Load JSON configuration
  • load_ingress_mapping(path) - Load YAML ingress mapping
  • load_env_region() - Load environment/region config
  • load_marker(cache_file) - Load test markers
  • save_marker(config, cache_file) - Save test markers

yaml_parser.lua~
YAML parsing utilities:
>
    local yaml_parser = require("yoda.yaml_parser")
    
    local data = yaml_parser.parse_ingress_mapping("ingress.yaml")
<

Configuration Hierarchy:~
1. User overrides (vim.g.yoda_config)
2. Environment variables
3. Configuration files (JSON/YAML)
4. Sensible defaults

QUALITY STANDARDS                               *yoda-architecture-quality*

Yoda.nvim maintains world-class quality standards across all modules.

Code Quality Metrics:~
  • Cyclomatic Complexity: <7 average
  • Function Length: <50 lines average
  • Module Size: <300 lines average
  • Test Coverage: ~95%
  • Zero code duplication (DRY)

Quality Gates:~
1. All code must pass linting (stylua)
2. All tests must pass
3. New code requires tests (>90% coverage)
4. Code review for all changes
5. Conventional commits required

Architectural Constraints:~
  • No circular dependencies
  • Utility modules (utils.lua, picker_handler.lua) have no plugin hard-dependencies
  • Domain modules use dependency injection where testability requires it
  • All public functions validate inputs

Development Standards:~
  • Use dependency injection for testability
  • Follow SOLID principles
  • Write self-documenting code
  • Include comprehensive error handling
  • Maintain backwards compatibility

Quality Assurance:~
  • Automated testing on CI
  • Code quality checks
  • Performance monitoring
  • Regular refactoring
  • Documentation updates
